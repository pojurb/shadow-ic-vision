# M001: Existing Thesis Loop

Status: `proposed`

Approval authority: user

---

## 1. User-Visible Outcome

The user can have a persistent, multi-turn conversation with the system to record an investment thesis for a specific public company (US or Indonesia). 
The system:
- Helps them decompose their raw thoughts into explicit, verifiable assumptions.
- Automatically conducts web research in the background (using official SEC/IDX feeds first, then falling back to explicitly cited crowdsourced/secondary sources with URLs and dates).
- Translates research findings into a structured briefing showing exact quotes, date, and a plain-language summary of what it means for each assumption.
- Proposes a smart Next Action (Call-to-Action) based on the evidence.
- Permanently stores the approved thesis, assumptions, and user-confirmed decision in a structured "Decision Library" that persists across sessions.

---

## 2. Scope and Non-Goals

### In Scope
- **Persistent Conversational UI:** The system maintains a running dialogue, acknowledging user input and offering gentle challenges to ground the user's assumptions.
- **Wedge Data Schema:** Storing a Thesis, its associated Assumptions (with status: *untested, verified, challenged, held-belief*), and Decisions.
- **Automated background research:** Crawling official sources (SEC, IDX) and falling back to reputable secondary sources.
- **Evidence Verification (Anti-Hallucination):** Character-exact substring validation for all quotes returned as evidence.
- **The Decision Library:** A structured local index of all past theses, assumptions, cited evidence, and recorded user actions.

### Out of Scope for M001
- Multi-user collaboration (Horizon 3).
- Brokerage integration, stock-price tracking, or return prediction.
- Passive/automated portfolio management.
- General unstructured web search beyond high-signal financial sources (to limit token bloat).

---

## 3. Workflows, States, and Recovery Behavior

### Workflow 1: Intake & Gentle Challenge
1. The user inputs a company (e.g., "PLTR") and starts describing their thesis in natural language.
2. The system gives a brief acknowledgment and asks a focused question to clarify or challenge a weak point.
3. If the user insists on an assumption despite the challenge, the system marks the assumption status as `held-belief` (unverified) and adds it to the tracking queue.
4. The conversation only concludes when the user explicitly clicks a confirmation CTA.

### Workflow 2: Automated Background Research
1. When an assumption is added, the system automatically triggers a web search task.
2. The system checks **Official Feeds** first (SEC EDGAR for US, IDX for Indonesia).
3. If no data is returned, the system falls back to **Secondary/Crowdsourced Feeds** (reputable financial blogs, verified aggregators) and extracts the exact text, URL, and publication date.
4. *Failure Recovery:* If no response is received from any feed, the system preserves the assumption, marks it `stale/unverified`, and flags the research gap.

### Workflow 3: Citation Verification (The Trust Engine)
1. For every piece of evidence generated by the LLM, the system runs a deterministic regex/exact-match check against the fetched source text.
2. If the quote is not a 100% exact substring of the source text, the system throws a hard warning, logs the error, and blocks the claim from being presented.
3. If verified, the system presents: **Quote + Source URL/Date + Plain-language impact summary**.

### Workflow 4: Smart Decision & Library Save
1. The system analyzes the collected evidence and prompts the user with a recommended decision CTA (e.g., *"Suggest updating assumption X based on recent 10-Q disclosures. Mark thesis as Investigate?"*).
2. The user confirms, modifies, or rejects the suggestion.
3. The final confirmed data is written to the local Decision Library.

## 4. Data Inputs, Outputs, and Persistence Rules

### Data Schema
- **Thesis:**
  - `id`: unique UUID
  - `ticker`: string
  - `company_name`: string
  - `market`: `US` | `ID`
  - `core_belief`: string
  - `status`: `active` | `archived`
- **Assumption:**
  - `id`: unique UUID
  - `thesis_id`: UUID
  - `statement`: string
  - `status`: `untested` | `verified` | `challenged` | `held-belief`
- **Evidence:**
  - `id`: unique UUID
  - `assumption_id`: UUID
  - `source_tier`: `official` | `secondary`
  - `source_name`: string
  - `source_url`: string
  - `publish_date`: string
  - `exact_quote`: string
  - `impact_summary`: string
- **Decision:**
  - `id`: unique UUID
  - `thesis_id`: UUID
  - `outcome`: `No Change` | `Investigate Further` | `Update Thesis` | `Archive`
  - `optional_action`: `Buy` | `Hold` | `Reduce` | `Exit` | `null`
  - `user_reasoning`: string
  - `timestamp`: string

### Persistence Rules
- Data must be stored in a structured, local, offline-first format.
- Schema changes must support backward compatibility.
- Export and deletion must target all tables cascade-linked to the user's ID.

---

## 5. Implementation Slices

- **Slice 1:** Conversational parser (CLI or simple page) that takes raw input, asks a clarifying question, and structures the Thesis/Assumptions in memory.
- **Slice 2:** Research crawler (fetches official financial SEC/IDX and fallback secondary pages) and passes them to the model.
- **Slice 3:** Strict citation checker (validates character-exact substrings in fetched HTML/markdown text).
- **Slice 4:** Decision prompt & persistent SQLite/JSON local library with export options.

---

## 6. Security and Provider Constraints
- User conversations and private investment theses must never be used to train external models.
- All outbound requests to LLMs or search engines must be logged and disclosed in the system console.

---

## 7. Evals & Acceptance Criteria

### Deterministic Tests
- **Verification Engine:** Mock HTML source inputs with exact quotes. Verify that the citation checker blocks any quote that has even a single character altered or added.
- **Persistence:** Verify that deleting a thesis cascades and deletes all linked assumptions, decisions, and evidence.

### Model Evals (Golden Dataset)
- We will construct 4 test cases representing typical user thesis entry (US and Indonesian equities).
- **Pass Thresholds:**
  - Citation Hallucination Rate: **0%** (Hard requirement).
  - Extraction completeness (identifying assumptions from user text): **>90%** match with expert PM labels.
  - Prioritization relevance (smart CTA alignment): **>80%** match with user reasoning.

---

## 8. Assumptions, Risks, and Explicit Deferrals
- **Assumption:** Access to SEC/IDX RSS and page structures remains stable or allows basic fallback scraping.
- **Risk:** High latency in the double-loop validation (fetching source ➔ parsing ➔ citation checking). Mitigated by running research async and keeping the user updated.
- **Deferral:** Automated push notifications, sector comparisons, portfolio tracking, and multiplayer workspaces are deferred to M002+.
